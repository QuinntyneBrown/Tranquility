# MDB subsystem — L2 requirements

## L2-MDB-001

- **Requirement:** Tranquility shall validate XTCE structural and reference integrity before activating a mission database.
- **Rationale:** Invalid mission models must be rejected before runtime processing.
- **Source:** OMG XTCE 1.3 (clause mapping pending: TBD-010)
- **Verification:** Test
- **Parent:** L1-MDB-001
- **Priority:** Must
- **Acceptance Criteria:**
  - GIVEN an XTCE document with broken references WHEN load is requested THEN activation is rejected with a validation report.

## L2-MDB-002

- **Requirement:** Tranquility shall expose mission database metadata through the documented MDB API resource model.
- **Rationale:** Operational clients need authoritative model metadata from the server.
- **Source:** External API documentation package (URL redacted)
- **Verification:** Test
- **Parent:** L1-MDB-001
- **Priority:** Must
- **Acceptance Criteria:**
  - GIVEN an active mission database WHEN MDB API methods are called THEN model counts and hierarchy data are returned.

## L2-MDB-003

- **Requirement:** Tranquility shall resolve parameters and commands by qualified names and aliases defined in XTCE.
- **Rationale:** Missions commonly use alias namespaces in operational tooling.
- **Source:** OMG XTCE 1.3 (clause mapping pending: TBD-010), External API documentation package (URL redacted)
- **Verification:** Test
- **Parent:** L1-MDB-001
- **Priority:** Must
- **Acceptance Criteria:**
  - GIVEN a parameter alias mapping WHEN a client uses alias-based identification THEN the same parameter is resolved as with qualified name.

## L2-MDB-004

- **Requirement:** Tranquility shall return explicit unsupported-feature diagnostics for XTCE constructs outside the approved support matrix.
- **Rationale:** Predictable model support boundaries are required for mission onboarding.
- **Source:** OMG XTCE 1.3, TBD-010
- **Verification:** Inspection
- **Parent:** L1-MDB-001
- **Priority:** Should
- **Acceptance Criteria:**
  - GIVEN an XTCE construct outside approved support WHEN load is requested THEN response identifies the unsupported construct and location.
