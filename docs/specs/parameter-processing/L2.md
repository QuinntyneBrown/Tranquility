# Parameter processing subsystem — L2 requirements

## L2-PAR-001

- **Requirement:** Tranquility shall compute engineering values from raw values using active calibration definitions.
- **Rationale:** Operators act on engineering units, not raw channel values.
- **Source:** OMG XTCE 1.3 (clause mapping pending: TBD-010)
- **Verification:** Test
- **Parent:** L1-PAR-001
- **Priority:** Must
- **Acceptance Criteria:**
  - GIVEN a raw telemetry sample with calibration metadata WHEN processed THEN output contains expected engineering value.

## L2-PAR-002

- **Requirement:** Tranquility shall evaluate limit states for processed parameters according to configured monitoring definitions.
- **Rationale:** Alarm validity depends on deterministic limit evaluation.
- **Source:** OMG XTCE 1.3, External API documentation package (URL redacted)
- **Verification:** Test
- **Parent:** L1-PAR-001
- **Priority:** Must
- **Acceptance Criteria:**
  - GIVEN a parameter value that crosses a configured threshold WHEN monitoring executes THEN resulting monitoring state reflects the configured level.

## L2-PAR-003

- **Requirement:** Tranquility shall publish parameter updates with documented acquisition and generation time fields.
- **Rationale:** Timeline and replay analysis require both timing dimensions.
- **Source:** External API documentation package (URL redacted), External API documentation package (URL redacted)
- **Verification:** Test
- **Parent:** L1-PAR-001
- **Priority:** Must
- **Acceptance Criteria:**
  - GIVEN a newly processed parameter WHEN update is emitted THEN payload includes `acquisitionTime` and `generationTime`.

## L2-PAR-004

- **Requirement:** Tranquility shall emit alarm transition notifications when a parameter monitoring level changes.
- **Rationale:** Downstream alerting depends on explicit state transitions.
- **Source:** External API documentation package (URL redacted)
- **Verification:** Demonstration
- **Parent:** L1-PAR-001
- **Priority:** Must
- **Acceptance Criteria:**
  - GIVEN a parameter transition from in-limits to out-of-limits WHEN processing runs THEN an alarm transition notification is observable through the API.
