# API subsystem — L2 requirements

## L2-API-001

- **Requirement:** Tranquility shall implement `GET /api/instances` with a response shape compatible with the documented List Instances method.
- **Rationale:** Instance discovery is a baseline operation for operational clients.
- **Source:** External API documentation package (URL redacted)
- **Verification:** Test
- **Parent:** L1-API-001
- **Priority:** Must
- **Acceptance Criteria:**
  - GIVEN a running Tranquility API WHEN a client calls `GET /api/instances` THEN the response includes an `instances` collection with documented instance fields.

## L2-API-002

- **Requirement:** Tranquility shall implement `POST /api/processors/{instance}/{processor}/commands/{name}` with request and response compatibility to the documented Issue Command method.
- **Rationale:** Command issue is a high-value interoperability path for existing control tooling.
- **Source:** External API documentation package (URL redacted)
- **Verification:** Test
- **Parent:** L1-API-001
- **Priority:** Must
- **Acceptance Criteria:**
  - GIVEN a valid command request WHEN a client posts to the documented Issue Command URI THEN the response contains command identity and assignment fields as documented.

## L2-API-003

- **Requirement:** Tranquility shall expose `/api/websocket` supporting documented `json` and `protobuf` subprotocol negotiation semantics.
- **Rationale:** Subscription clients rely on the documented WebSocket handshake and subprotocol selection.
- **Source:** External API documentation package (URL redacted)
- **Verification:** Test
- **Parent:** L1-API-001
- **Priority:** Must
- **Acceptance Criteria:**
  - GIVEN a WebSocket upgrade request WHEN the client requests `Sec-WebSocket-Protocol: json` THEN the session is accepted with JSON message encoding.
  - GIVEN a WebSocket upgrade request WHEN the client requests `Sec-WebSocket-Protocol: protobuf` THEN the session is accepted with protobuf message encoding.

## L2-API-004

- **Requirement:** Tranquility shall return non-2xx API failures in the documented exception envelope format.
- **Rationale:** Error handling in existing clients is keyed to the documented exception schema.
- **Source:** External API documentation package (URL redacted)
- **Verification:** Test
- **Parent:** L1-API-002
- **Priority:** Must
- **Acceptance Criteria:**
  - GIVEN an invalid API request WHEN Tranquility returns an error THEN the body contains `exception.type` and `exception.msg`.

## L2-API-005

- **Requirement:** Tranquility shall encode API timestamps in UTC ISO 8601 / RFC 3339 format.
- **Rationale:** Time handling consistency is required for timeline and replay interoperability.
- **Source:** External API documentation package (URL redacted)
- **Verification:** Inspection
- **Parent:** L1-API-002
- **Priority:** Must
- **Acceptance Criteria:**
  - GIVEN any response containing time fields WHEN the payload is serialized THEN timestamp strings are UTC ISO 8601 / RFC 3339.
